# Decision journal — 27


## Decision
Determine "unsupported" at report-generation time by calling `analysis.ForLanguage(lang) == nil`. The analysis package already has a registry (`analysis.Languages()` returns all supported, `analysis.ForLanguage()` returns nil for unsupported). Currently registered: go, ts, js, py, rs. No `Unsupported` field needs to be added to `CertificationRecord` — the report package derives it from the language identifier in the UnitID. This respects the non-goal of not changing scorer.go or any engine code.

## Decision
The mechanism for detecting unsupported units is `analysis.ForLanguage(record.UnitID.Language()) == nil`. The `analysis` package's registry is populated by `init()` functions in `go_analyzer.go` (go), `ts_analyzer.go` (ts, js), `python_analyzer.go` (py), and `rust_analyzer.go` (rs). Any language NOT in this registry — e.g., swift, shell/sh, ruby/rb, java, cpp, etc. — returns nil from ForLanguage and is therefore unsupported.

There is NO existing `Unsupported` field on `CertificationRecord` or `Grade`. The issue's mention of `Grade != "N/A"` and `Unsupported==true` is aspirational — the actual partitioning must use `analysis.ForLanguage`. The non-goals mention "Do NOT change scorer.go or language_tiers.go" but these files do not exist in the current codebase; they may be planned for future work. The actual target files are: card.go, badge.go, health.go, detailed.go, full.go, report_tree.go, and their tests.
